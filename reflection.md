1. CSRF and the state Parameter: In your own words, explain how an attacker could perform a Cross-Site Request Forgery (CSRF) attack on an OAuth flow. How does using the state parameter, as recommended, prevent this specific attack?
- A Cross-Site Request Forgery (CSRF) attack in an OAuth flow can happen when an attacker tricks a user’s browser into sending or accepting an OAuth authorization request without the user realizing it. In this type of attack, the attacker may try to link the victim’s browser session to the attacker’s own authorization code or account. For example, the attacker could start an OAuth request using their own account and then trick the victim into clicking a malicious link. If the application does not properly verify the request, the victim’s browser could unknowingly complete the OAuth process using the attacker’s authorization code. This could result in the victim being logged into the attacker’s account or the attacker gaining unauthorized access to information connected to the OAuth flow. The state parameter helps prevent this attack by maintaining a unique value between the initial authorization request and the callback response. When the OAuth process begins, the application generates a random state value and stores it temporarily. After the user authenticates, the OAuth provider sends the same state value back with the authorization response. The application then checks whether the returned value matches the original one. If the values do not match, the request is rejected because it may have been altered or forged by an attacker. This adds an important layer of protection by ensuring the response belongs to the legitimate user request and not a malicious third-party attempt.

2. Redirect URI Attacks: The article mentions that validating a redirect_uri by simply checking the domain or allowing subdomains is a common mistake. Describe a hypothetical scenario where a “leaky” redirect_uri validation (e.g., one that allows any path on a valid domain) could be exploited to steal an authorization code.
- A “leaky” redirect_uri validation issue can happen when an OAuth server only checks if the redirect URL belongs to a trusted domain instead of checking the exact URL. For example, imagine ExampleCorp allows any redirect that starts with https://app.example.com/. That might seem safe at first, but it becomes risky if users can upload files or create pages on that domain. An attacker could upload a fake page like https://app.example.com/uploads/evil.html that contains hidden JavaScript made to steal authorization codes. The attacker could then send a victim an OAuth login link using that page as the redirect URI. Since the OAuth server only checks the domain name and sees app.example.com, it accepts the request. After the victim logs in, they get redirected to the attacker’s page with the authorization code included in the URL. The malicious script on the page can read that code and send it straight to the attacker without the victim realizing it happened. The attacker could then exchange the stolen code for an access token and possibly gain access to the victim’s account or personal information. This is why OAuth systems should always require exact redirect_uri matches instead of allowing any path or subdomain on a trusted domain.

3. User Experience vs. Security: Adding a third-party login option like “Login with Google” is a significant user experience improvement. However, it also introduces complexity and new potential security risks. Based on the article and your own thoughts, describe one key trade-off a development team must consider when deciding to implement OAuth. (For example, think about the balance between convenience for the user and the responsibility of the application to protect user data).
- One major trade-off a development team has to think about when implementing OAuth is balancing convenience for users with the added security responsibilities that come with relying on a third-party login system. Features like “Login with Google” make signing in much faster and easier because users do not have to create another password or remember new login information. This can improve user experience and even encourage more people to register for the app. However, adding OAuth also increases the complexity of the application and creates new security risks if it is not implemented correctly. For example, mistakes involving redirect_uri validation, token storage, or authorization code handling could allow attackers to steal access tokens or user information. The development team also has to trust the third-party provider to securely manage authentication and protect user data. In addition, if the provider experiences downtime or security issues, it could affect users trying to access the application. While OAuth can improve both convenience and security compared to weak passwords, developers must carefully design and maintain the system to make sure the added convenience does not create vulnerabilities that put user accounts or personal information at risk.
